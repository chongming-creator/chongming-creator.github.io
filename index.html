分布式和伪分布式

分布式模式下，集群启动多个服务，JobTracker管理job，NameNode管理HDFS，TaskTracker管理每个工作节点上执行的job task，DataNode管理每个节点上存放的分布式文件系统中的文件数据块

下图展示了典型的分布式集群配置，其实伪分布式也差不多相同，它实际上是一个单节点的集群

可以将*.domain.pvt作为集群内外网DNS命名规范

用户可配置的一个属性是存储表的顶级目录，由属性hive.metastore.warehouse.dir指定，在Apache Hadoop和MapR分钟中默认是/user/hive/warehouse，当Hadoop配置的是分布式或伪分布式模式时，这个路径被认为是分布式文件系统中的路径

数据类型和文件格式

基本数据类型

数据类型 长度 示例
TINYINT 1byte有符号整数 20
SMALINT 2byte有符号整数 20
INT 4byte有符号整数 20
BIGINT 8byte有符号整数 20
BOOLEAN 布尔类型 true
FLOAT 单精度浮点数 3.14
DOUBLE 双精度浮点数 3.14
STRING 字符序列，可指定字符集，使用单引号/双引号 “hello world”或’hello world’
TIMESTAMP 整数，浮点数或字符串 1327882394（秒）；1327882394.123456789（秒.纳秒）；’2019-9-30 12:00:00.123456789’
BINARY 字节数组
  1. 所有数据类型都是对java中的接口的实现，因此行为细节和java中的对应类型完全一致
  2. 与关系型数据库不同，没有限制最大长度的”字符数组“，hive根据不同字段间的分隔符来判断，强调优化磁盘读写性能，限制列值长度相对来说不重要
  3. BINARY和很多关系型数据库的VARBINARY数据类型类似，和BLOB类型不同（BINARY的列是存储在记录中的），BINARY可以在记录中包含任意字节，这样可以防止Hive尝试将其作为数字、字符串等进行解析
  4. 当用户比较表中不同类型的记录时（如float和double比较），Hive会隐式地将类型转换成两个类型中较大的那个，再进行同类型比较，也可以使用手动转换，使用函数cast（s as int）

集合数据类型

数据类型 描述 语法示例
STRUCT 类似C语言的结构体，如：STRUCT<street:STRING, city:STRING, state:STRING> struct(‘五常’,’杭州’,’拥堵’)
MAP 键-值对元素集合，可以通过key获取到value map(‘first’,’join’,’last’,’doe’)
ARRAY 一组具有相同类型和名称的变量的集合，可以通过下标获取元素 Array(‘john’,’doe’)
  1. 集合数据类型的名称是保留字
  2. 优点：大多数关系型数据库不支持集合类型，在大数据系统中，不遵循标准格式的好处是可以提高更高吞吐量的数据，可以减少头部寻址来增加查询次数
  3. 缺点：然而破坏标准格式会增加数据冗余的风险，从而导致消耗不必要的磁盘空间，还有可能导致数据不一致，当数据发生改变时冗余的数据可能存在无法进行同步的风险

读时模式

传统数据库是写时模式，也就是说在数据写入数据库时进行检查

hive则是在查询时对数据进行验证，也就是读时模式

读时模式的具体做法是hive在读取数据时，如果发现存在不匹配的字段，则对这些字段返回null值

HiveQL - 数据定义

hiveql是hive的查询语言，在所有近亲语言中和mysql方言最像，但是又有一定的差别。hive不支持行级插入操作、更新操作和删除操作，也不支持事务。

Hive中的数据库

hive中的数据库本质上是表的目录或者命名空间，在多用户的集群下，可以有效避免表的命名冲突

hive会为每个数据库创建一个目录，数据库中的表将会以这个数据库目录的子目录形式存储（default除外，这个数据库本身没有自己的目录）数据库所在目录位于hive.metastore.warehouse.dir所指定的顶层目录之后，假设用户使用默认配置，目录就是/user/hive/warehouse，我们创建test库时就会有一个/user/hive/warehouse/test.db目录

主要参考资料：谁说深入浅出虚拟机难？现在我让他通俗易懂（JVM）

简单地做下jvm的学习笔记

概念

Jvm

java虚拟机（java virtual machine）缩写，java使用jvm屏蔽与平台相关的信息，做到平台无关性。jvm执行字节码时把字节码解释成具体平台上的机器指令执行

JDK

java语言软件开发工具包（java development kit）（注：软件开发工具包software development kit）

jdk包含：javac、jar、Javadoc、jdb、java、appletviewer、javah、javap、jconsole

JRE

java运行环境（java runtime environment）运行java所需的环境集合，包括：java runtime environment（java平台，不包含jdk）和java plug-in（用来跑java小程序）

JVM运行时数据区

基本结构：

运行时结构：

从上图可以看出，程序计数器、java虚拟机栈、本地方法栈为线程私有，生命周期与线程同

  1. 程序计数器

    当前线程所执行的字节码行号指示器

    独立存储，cpu多线程机制，为了线程切换后的复位

    记录正在执行的java方法的字节码指令地址，如果执行的是native方法，计数器值为空

  2. java虚拟机栈

    描述java方法执行的内存模型，每个方法执行的时候创建栈帧存储局部变量表、操作数栈、动态链接、方法出口等信息，每个方法从被调执行至完成就对应一个栈帧在虚拟机栈中从入栈到出栈的过程

  3. 本地方法栈

    与虚拟机栈类似，为native方法服务

java堆（Java Heap）与方法区（Method Area）为共有

  1. java堆（GC堆）

    所有线程共享

    虚拟机启动时创建

    存放对象实例（几乎所有对象实例及数组）

    采用分代收集算法（java堆分为新生代和老年代，新生代又有Eden、From survivor、To Survivor）

    可以处于不连续的内存空间，可扩展（通过控制-Xms和-Xmx）

  2. 方法区

    所有线程共享

    虚拟机启动时创建

    存放已被虚拟机加载的类信息、常量、静态变量、JIT编译后的代码等

    不需要连续的内存空间，可扩展、可固定大小，可不实现垃圾收集

    运行时常量池：

    方法区的一部分，存放编译期生成的各种字面量和符号引用，详见java虚拟机：运行时常量池

直接内存

不是虚拟机运行时数据区的一部分，也不是jvm规范中定义的内存区域，这部分内存也被频繁使用

JVM内存模型（JMM）

java1.8之后移除了永久带，引入Metaspace（元空间）

传送门：Java8 对应的 JMM 划分

JMM中的并发编程

两个概念

  1. 线程间通信

    线程间可通过共享内存和消息传递两种方式来进行。

    共享内存：线程间通过读写内存中的公共状态来隐式通信，例如通过共享对象进行通信

    消息传递：线程之前没有公共状态，通过发送消息来进行显式通信，例如wait()和notify()

  2. 线程间同步

    程序用于控制不同线程之间操作发生相对顺序的机制

    共享内存模型：同步显式进行，通过指定某个方法或代码段需要在线程间互斥执行

    消息传递模型：消息的发送在接收之前，因此为隐式进行

Java内存模型

概念：jmm定义线程和主内存之间的抽象关系是线程之间的共享变量存储在主内存中，每个线程都有一个私有的本地内存，本地内存中存储该线程读写共享变量的副本（本地内存是一个抽象概念，不真实存在）

JVM对java内存模型的实现：

线程栈：

  1. 存储当前方法的所有本地变量
  2. 所有原始类型的本地变量
  3. 本地变量是一个对象时存放其引用
  4. 一个对象的成员方法包含的本地变量
  5. 一个对象的成员变量

堆：

  1. 本地变量是一个对象时存放对象本身
  2. java应用创建的所有对象信息（不管对象是成员变量还是本地变量）
  3. static变量以及类本身相关信息

jvm桥接硬件架构

jmm和硬件内存架构是交叉关系，由此产生两个问题

  1. 共享对象的可见性：

    多线程操作同一共享对象时，在没有并发控制的情况下，一个线程的共享对象的更新可能导致其他线程看不见

    可以使用volatile，变量直接从主存读取，更新也直接写到主存。基于Memory Barrier实现

  2. 竞争现象：

    多个线程同时修改同一共享对象

    使用synchronized关键字，保证同一个时刻只能有一个线程进入代码竞争区。

  3. volatile和 synchronized区别

    • volatile本质是在告诉jvm当前变量在寄存器（工作内存）中的值是不确定的，需要从主存中读取； synchronized则是锁定当前变量，只有当前线程可以访问该变量，其他线程被阻塞住。
    • volatile仅能使用在变量级别；synchronized则可以使用在变量、方法、和类级别的
    • volatile仅能实现变量的修改可见性，不能保证原子性；而synchronized则可以保证变量的修改可见性和原子性
    • volatile不会造成线程的阻塞；synchronized可能会造成线程的阻塞。
    • volatile标记的变量不会被编译器优化；synchronized标记的变量可以被编译器优化

支撑jmm的基础原理

  1. 指令重排序：编译器在不改变单线程程序语义的前提下重新安排语句的执行顺序；如果不存在数据依赖性，处理器可以改变语句对应机器指令的执行顺序
  2. 内存屏障（memory barrier）：禁止特定类型的处理器重排序

java动态代理机制

2019-09-16

关于设计模式-代理模式：先上一段百科中查到的代理模式的定义：为其他对象提供一种代理以控制对这个对象的访问。在某些情况下，一个对象不适合或者不能直接引用另一个对象，而代理对象可以在客户端和目标对象之间起到中介的作用。

也就是说我们通过代理对象来访问实际对象，代理模式就是在访问实际对象时引入一定程度的间接性，因为这种间接性，可以附加多种用途。

通俗地讲，这就像我们生活中的中介，我们只需要挑选我们想要的东西并付钱，中间其他一系列复杂流程委托给中介来完成就好了。

代理模式的三种实现方式

静态代理

定义一个Worker接口，用WorkerImpl类来实现这个接口

定义一个代理类，同样实现Worker接口，并持有WorkerImpl类的引用worker，实现接口内的方法，并在调用worker的doWork方法前后完成一些工作。

使用：创建WorkerImpl对象，然后把对象作为构造参数创建代理类对象WorkProxy，最后通过代理类对象WorkProxy来进行业务方法doWork调用。

静态代理的局限性：一个代理类只能对一个业务接口的实现进行包装，当有多个业务接口的时候必须用多个代理类一一包装，如果多个业务方法的预处理操作和调用后操作相同，则会产生很多重复代码。

JDK动态代理

jdk动态代理使用反射机制，在程序调用到代理类对象时jvm根据传进来的业务实现类对象以及方法名 ，动态地创建代理类的class文件并被字节码引擎执行，然后通过该代理类对象进行方法调用。

Worker接口和实现类仍然使用上面的

创建动态代理类DynamicProxy实现InvocationHandler接口并重写invoke方法，在invoke方法中通过反射拿到doWork方法，并在doWork前后进行一些工作。

使用：创建动态代理对象dynamicProxy，调用getProxy方法获取到代理类对象，通过代理类对象调用doWork方法。

Jdk动态代理的局限性：需要业务类实现接口，在获取代理方法时需要根据方法内的接口名进行调用，因此只有接口中有的方法，可以被代理。详见java动态代理探究

Restful风格最佳实践

REST来历

Roy Thomas Fielding在他2000年的博士论文中提出了REST的概念，Representational State Transfer这个词，翻译成中文大意为表现层状态传输。由于他是HTTP协议（1.0版和1.1版）的主要设计者、Apache服务器软件的作者之一、Apache基金会的第一任主席，所以REST原则迅速流行起来。当一个软件架构符合REST原则，我们称之为RESTful架构。

为什么要使用REST？

众所周知，在桌面时代，网页是前后端融合在一起的，但是随着移动互联网的发展，各种各样类型的client层出不穷，RESTful可以通过一套统一的接口为WEB、IOS、Android提供服务。而且对于一些平台来说，如微信公众号、微博开放平台、Facebook platform等，它们不需要显式的前端，只需要一套提供服务的接口，这样来说RESTful是最好的选择。

相关概念

在了解REST之前我们先来了解几个概念：

幂等性：在HTTP/1.1规范中幂等性的定义是如果方法调用一次和多次产生额外的效果是相同的，它就具有幂等性。例如，在HTTP中使用GET方法通常用于从服务器获取资源，无论调用多少次产生的额外效果都是从服务器获取资源，所以GET具有幂等性；而POST方法通常用于提交数据在服务器上创建一个资源，由于最终创建的结果每次都是不同的，所以POST不具有幂等性；但是PUT方法却是幂等的，因为每次调用产生的效果都是对资源进行更新。

安全性：安全方法是指不修改资源的 HTTP 方法。更准确来说是：不改变资源的表示形式。对于安全方法，它仍然可能改变服务器上的内容或资源，但这必须不导致不同的表现形式。

在啰嗦了这么多之后终于可以进入主题了，接下来让我们真正地走近REST。

什么是REST?

REST全称Representational State Transfer，即表现层状态转移，乍看之下不明觉厉，之所以晦涩难懂是因为去掉了主语，应该是Resource Representational State Transfer，通俗来讲就是：资源在网络中以某种表现形式进行状态转移。
拆开理解：

Resource：资源，即数据（前面说过网络的核心）。比如 newsfeed，friends等；Representational：某种表现形式，比如用JSON，XML，JPEG等；

State Transfer：状态变化。通过HTTP动词实现。

RESTful API设计规则

  1. 应将api部署在专用域名下

  2. URL区分大小写，/Posts和/posts是两个不同的URL，指向不同的资源

  3. 使用连接符“-”，而不使用下划线“_”，避免和浏览器中超链接（文字并附带下划线）重合，影响可读性

  4. 参数列表要encode

  5. URL中不应该出现动词，动词应该用HTTP方法表示

  6. URL中名字表示资源集合，建议使用复数形式

  7. 虽然在URL中使用“/”表达层级，但是为了避免层级过深，适当用参数表示，例如：GET /customers/{customerId}/cars?offset=10&limit=5

Request

通过标准HTTP方法对资源CRUD

  • GET：查询资源

  • POST：在服务器新建一个资源

  • PUT：在服务器更新资源

  • DELETE：从服务端删除资源

Response

  • 采用JSON

  • 默认情况下JSON外层不需要嵌套大括号，API需要支持JSONP跨域访问或者客户端无法访问HTTP header才需要加上嵌套大括号

  • 默认情况下不要过滤API输出中的空格，并且要支持gzip

API版本控制

一般采用在URL中存放，例如：POST/v4/customers.json

过滤信息

如果记录数量过多，API应该提供参数，过滤请求结果，例如：?page=2&per_page=100

HTTP状态码

  • 200 OK - 对成功的GET、PUT、PATCH或DELETE操作进行响应。也可以被用在不创建新资源的POST操作上

  • 201 Created - 对创建新资源的POST操作进行响应。应该带着指向新资源地址的Location header)

  • 204 No Content - 对不会返回响应体的成功请求进行响应（比如DELETE请求）

  • 304 Not Modified - HTTP缓存header生效的时候用

  • 400 Bad Request - 请求异常，比如请求中的body无法解析

  • 401 Unauthorized - 没有进行认证或者认证非法。当API通过浏览器访问的时候，可以用来弹出一个认证对话框

  • 403 Forbidden - 当认证成功，但是认证过的用户没有访问资源的权限

  • 404 Not Found - 当一个不存在的资源被请求

  • 405 Method Not Allowed - 所请求的HTTP方法不允许当前认证用户访问

  • 410 Gone - 表示当前请求的资源不再可用。当调用老版本API的时候很有用

  • 415 Unsupported Media Type - 如果请求中的内容类型是错误的

  • 422 Unprocessable Entity - 用来表示校验错误

  • 429 Too Many Requests - 由于请求频次达到上限而被拒绝访问

认证

RESTful API应该是无状态。这意味着对请求的认证不应该基于cookie或者session。相反，每个请求应该带有一些认证凭证。

  • 使用SSL的情况下，认证凭证可以简单的使用随机生成的access token，把其做为HTTP Basic Auth中user name字段的值传给API。

  • 如果用户无法从API管理后台获取到token，则使用OAuth 2（也是基于SSL）来把安全token传递给第三方

  • 支持JSONP的API可以使用一个额外的查询参数access_token，因为JSONP的请求没法发送HTTP Basic Auth凭证或者Bearer token。

传参

  1. 放在url路径中，例如：GET/v1/customers/{customerId}.json

  2. 为避免层级过深，适当使用参数表示，例如：GET/v1/customers.json?ids=&uids=&search=&shopCode=&siteId=

  3. 以JSON形式放在url路径中，例如：http://localhost:8080/app/names?queryDtoStr={“query1”:12,”query2”:2,”query3”:2}

  4. POST请求将参数以JSON的形式放在body中， 服务端使用@RequestBody注解将其解析为对象。

  1. 2004年，谷歌发表三篇论文：分布式文件系统GFS、大数据分布式计算框架MapReduce、NoSql数据库系统BigTable，后称”三驾马车“，打开了分布式处理大数据的大门
  2. 2006年，天才程序员Doug Cutting将大数据相关模块从其开发的开源搜索引擎Nutch中独立出来，形成新的项目–Hadoop，主要包括HDFS（分布式文件系统）和MapReduce（大数据计算引擎）
  3. 2007年，除Yahoo之外，百度和阿里巴巴开始使用Hadoop处理大数据存储和计算工作
  4. 2008年，Hadoop正式成为Apache顶级项目，Doug Cutting成为Apache基金会主席，同年，Cloudera成立；同年9月，Yahoo发布Pig代替MapReduce完成复杂编程，之后FaceBook发布Hive，支持使用SQL语法进行大数据计算
  5. 2011年，经统计，截止当年FaceBook大数据平台上的90%的作业来源于Hive，同时，伴随Hadoop周边产品出现（支持数据导入导出的Sqoop、针对大规模日志进行分布式收集的Flume、MapReduce工作流调度引擎Oozie，以及NoSql相关的Hbase、Cassandra等），大数据生态体系逐渐形成
  6. 2012年，Yarn成为独立项目开始运营，成为大数据平台上最主流的资源调度系统；同年，UC伯克利AMP实验室开发的Spark开始崭露头角，内存成为数据运行中的主要存储介质，减少了MapReduce执行Map和Reduce计算重启作业带来的开销。除离线计算领域发展之外，在实时计算领域，Storm、Flink等流式计算框架纷纷出现，其中Flink可以同时支持流式计算和批处理计算
  7. 2013年，在数据量的爆发式增长，与框架技术的逐渐完善，大数据形成社会浪潮，影响生活的方方面面。有媒体将这一年称为”大数据元年“

Executors线程池工厂

  1. 固定线程池

    创建固定线程数量的线程池，新任务提交时有空闲线程则执行，无空闲线程则新任务暂存在任务队列，待有空闲线程时处理

  2. 单例线程池

    创建只有一个线程的线程池，多余任务保存在任务队列（默认无界，上限Integer.MAX_VALUE），待有空闲线程时处理

  3. 缓存线程池

    创建可根据实际情况调整线程数量的线程池，线程数量不确定，有空闲线程时优先复用，所有线程都在工作，如果有新任务提交，则创建新的线程处理任务

  4. 任务调用线程池

    创建ScheduledThreadPoolExecutor对象，可以指定线程数量，三个方法：

    • scheduleAtFixedRate 参数：command线程指令，initialDelay第一次延迟时间，period任务间隔时间，unit时间单位
    • scheduleWithFixedDelay 参数： command线程指令，initialDelay第一次延迟时间，delay（公式：delay+任务执行时间=间隔时间），unit时间单位
    • schedule， command线程指令，delay延迟时间，unit时间单位

手动创建线程池

  • 线程队列：

    线程池默认提供四个队列

    • 无界队列： 默认大小 int 最大值，因此可能会耗尽系统内存，引起OOM，非常危险。
    • 直接提交的队列 ： 没有容量，不会保存，直接创建新的线程，因此需要设置很大的线程池数。否则容易执行拒绝策略，也很危险。
    • 有界队列：如果core满了，则存储在队列中，如果core满了且队列满了，则创建线程，直到maximumPoolSize 到了，如果队列满了且最大线程数已经到了，则执行拒绝策略。
    • 优先级队列：按照优先级执行任务。也可以设置大小。

  • 拒绝策略：

    jdk默认提供四种拒绝策略

    • AbortPolicy ：直接抛出异常，阻止系统正常工作.
    • CallerRunsPolicy : 只要线程池未关闭，该策略直接在调用者线程中，运行当前被丢弃的任务。显然这样做不会真的丢弃任务，但是，任务提交线程的性能极有可能会急剧下降。
    • DiscardOldestPolicy: 该策略将丢弃最老的一个请求，也就是即将被执行的一个任务，并尝试再次提交当前任务.
    • DiscardPolicy: 该策略默默地丢弃无法处理的任务，不予任何处理，如果允许任务丢失，我觉得这是最好的方案.

    注：可以自己实现拒绝策略，实现 RejectedExecutionHandler 接口，并重写 rejectedExecution 方法即可

  • 线程工厂：

    需要修改线程工厂生产线程的细节，实现 ThreadFactory 接口，重写 newThread 方法即可。可以使用google 的 guaua 提供的 ThreadFactoryBuilder 工厂。可以自定义线程名称，是否守护，优先级，异常处理等等，功能强大。

  1. 当线程池线程数<核心线程数时，只要来任务就新建线程
  2. 当线程池线程数>核心线程数，且<最大线程数时
